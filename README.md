# Meeting Calendar (MVP)

Внутренний корпоративный сервис **«Календарь встреч компании»** для тестового задания AI-специалиста.  
Проект работает **только на фронтенде** и сохраняет данные в `localStorage` (без backend и сторонних сервисов).

## Назначение

Сотрудники могут:

- просматривать встречи;
- создавать встречи;
- видеть занятость коллег (конфликты по времени участников);
- добавлять новых сотрудников.

Все изменения сохраняются в браузере пользователя через `localStorage`.

## Функциональность MVP

### Главная страница

- Верхняя панель:
  - название `Meeting Calendar`;
  - описание: *«Внутренний календарь встреч компании»*;
  - кнопки:
    - `+ Добавить встречу`
    - `+ Добавить сотрудника`
- Режим просмотра:
  - `День` / `Неделя`
  - выбор даты
- Фильтры:
  - `Все сотрудники ▼` (показывает встречи только выбранного сотрудника)
  - поиск по названию встречи
- Демоданные:
  - `Очистить демо-данные`
  - `Создать демо-команду` (появляется после очистки)

### День / Неделя

- **День**: список встреч выбранной даты, карточки отсортированы по времени старта.
- **Неделя**: 7 колонок (понедельник–воскресенье), внутри каждого дня карточки отсортированы по времени старта.

### Карточка встречи

В карточке отображается:

- цветная полоска (визуальный акцент);
- время `startTime - endTime`;
- название встречи;
- список участников;
- описание;
- кнопка `Удалить`.

### Добавление сотрудника

В отдельном модальном окне:

- Имя
- Фамилия
- Должность

Удаление сотрудника доступно **только если у него нет активных встреч** (встречи с `endTime`, которые ещё не завершились относительно текущего времени).

### Добавление встречи и проверка конфликтов (real-time)

В модальном окне создания встречи:

- Название встречи
- Дата
- Время начала
- Время окончания
- Участники (множественный выбор)
- Описание

При изменении `Дата / Время начала / Время окончания / Участники` выполняется **моментальная проверка конфликтов**:

- если выбранный сотрудник уже участвует в другой встрече в пересекающийся интервал времени — возле его имени появляется красная метка `занят`;
- показывается сообщение вида:
  - `Анна Петрова уже участвует во встрече с 14:00 до 15:00`
- кнопка `Создать встречу` блокируется до устранения конфликта.

## Архитектура (как устроено)

### Состояние и данные

- Все доменные данные (сотрудники и встречи) хранятся в состоянии `App`.
- Состояние персистится в `localStorage` под ключом `meeting-calendar:v1`.
- При первом запуске автоматически генерируются демо-сотрудники (8–10 человек) и демо-встречи.
- После `Очистить демо-данные` данные сбрасываются в пустое состояние, а `seed` больше не запускается автоматически — появляется кнопка `Создать демо-команду`.

Ключевые файлы:

- `src/lib/storage.ts` — загрузка/сохранение в `localStorage`, seed демо-данных
- `src/lib/meetingConflicts.ts` — логика поиска конфликтов по пересечению интервалов времени
- `src/components/dialogs/MeetingDialog.tsx` — форма создания встречи + real-time конфликты
- `src/components/dialogs/EmployeeDialog.tsx` — форма добавления сотрудника + удаление с проверкой активных встреч
- `src/components/views/DayView.tsx` и `src/components/views/WeekView.tsx` — отображение встреч

### Представления

- **DayView** фильтрует встречи по `date === selectedDateISO`.
- **WeekView** вычисляет понедельник недели и строит 7 ISO-дней; затем группирует встречи по дню.

### Конфликты

- Пересечение интервалов считается на пересечении полуинтервалов:
  - `aStart < bEnd && aEnd > bStart`
- Конфликт ищется только для встреч на той же календарной дате.

## Установка и запуск

Выполните из папки проекта:

```bash
npm install
npm run dev
```

Открыть в браузере:

- `http://localhost:5173`

## Сборка

```bash
npm run build
```

## Деплой на GitHub Pages

Сборка выводится в папку `docs/` (источник для GitHub Pages).

```bash
npm run deploy
```

В настройках репозитория: **Settings → Pages → Build and deployment → Source: Deploy from a branch → Branch: `main` / Folder: `/docs`**.

Сайт: https://anna-dev82.github.io/meeting-calendar/

## Важно про MVP (ограничение данных)

В рамках MVP данные сохраняются в `localStorage` браузера.  
В рабочей версии вместо `localStorage` должна использоваться централизованная база данных и корпоративная авторизация сотрудников.
